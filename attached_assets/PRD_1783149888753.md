# 1. Overview

NRZ International (est. 2008) is a Rawalpindi-based visa consultancy and education agency specializing in university admissions in China (MBBS, BDS, Clinical Medicine, Nursing, Pharmacy, Engineering) and Chinese-language courses. This document defines the requirements for a **website platform** carrying the full functionality of the NRZ International system — the same feature set as the companion mobile app, accessible to students, visa consultants, and administrators from a browser.

The platform's purpose is to move the company's current paper-based application and document-review process (visible in its printed brochures and admission forms) into a structured, trackable digital workflow, without changing how payments are actually collected (in person / bank transfer, outside the platform).

---

# 2. User Roles

| Role | Access |
|---|---|
| **Student** | Self-registers an account. Browses universities/programs, submits applications, uploads documents, tracks status, receives notifications, contacts their assigned consultant. |
| **Consultant** | Each consultant is assigned a fixed set of students (**one consultant per student**, owning that student's entire journey from inquiry to visa completion). Reviews documents, approves/rejects/requests changes, manages internal audit documents, sends feedback messages, marks payment received. |
| **Admin** | Full oversight: sees every student across all consultants, manages universities/programs/courses/scholarships, manages dynamic document requirements, views system-wide activity logs, reassigns consultants, sees dashboard summary metrics. |

Roles are enforced at the database level (row-level security), not just hidden in the UI — a student can never access another student's data, and a consultant can never access a student who isn't assigned to them.

# 3. Student Flow

## 3.1 Browse
On login, the student sees a browsable catalog of universities and programs, filterable by program type (MBBS, BDS, Clinical Medicine, BS Nursing, Pharmacy, Engineering disciplines, Chinese Language Course) and by province/city.

## 3.2 University / Program Detail
Selecting a university opens a detail page containing everything currently printed on NRZ's brochures:

- University name, city, province, founding year, PMDC listing status
- "Why choose this university" — recognitions (WHO, FAIMER, ECFMG, PLAB, WCAME, UNESCO), graduate batch counts, current student population, HSK requirement (or "No HSK in first year"), hostel/food notes
- **Every program offered at that university**, each as its own card, since fees and requirements differ by program even within the same school (e.g. Chifeng University separately offers BDS, Clinical Medicine, and BS Nursing, each with different fees and durations)
- Full fee breakdown per program: 1st-year fee (itemized: university, hostel, application, admission, development charges), 2nd-year-onward fee (university fee + hostel fee, then total/year), and the one-time Pakistan-side charge (JW202 + company service fee)
- Program notes: age eligibility range, minimum academic percentage required, visa extension / medical checkup / medical insurance costs, currency-fluctuation and miscellaneous-expense disclaimers
- Downloadable brochure PDF
- Available scholarships for that university (e.g. top-15 scholarships, government scholarships, unconditional bachelor scholarships)

## 3.3 Apply
An **Apply** button on each specific program starts that program's application. Because required documents differ by program (medical programs vs. engineering vs. Chinese language course each have distinct document sets), the document checklist shown is pulled dynamically per program, not hardcoded.

## 3.4 Three-Phase Application Process

**Phase 1 — Pre-Admission Documents (mandatory)**
Basic admission documents: SSC result card/certificate, HSSC result card/certificate (or Hope Certificate from college), ID card copy, photographs (white background), plus any university-specific requirements. This phase must be fully approved before the application can proceed.

**Payment Checkpoint (between Phase 1 and Phase 2)**
Once the consultant has reviewed and is satisfied with Phase 1 documents, they send the student an in-app feedback message (e.g. "please proceed to payment"). Payment itself happens off-platform (in person / bank transfer). The consultant then marks a **Payment Received** flag once confirmed. Only after this flag is set can the consultant formally approve Phase 1 and move the application into Phase 2. This checkpoint is a required gate in the application state machine, not a UI suggestion.

**Phase 2 — JW202 / Visa Invitation Letter Documents (mandatory)**
Unlocked after Phase 1 approval and receipt of the pre-admission letter. Documents: passport copy, police clearance certificate (since birth to present), physical examination (medical checkup), current bank statement (last 6 months), introduction video. Reviewed and approved by the consultant before proceeding.

**Phase 3 — Visa Documents (optional / informational)**
Guidance and reminders for documents needed at the visa appointment stage. This phase does not block progress — it is informational, helping the student prepare ahead of time.

## 3.5 Change Requests
At any review step, if a submitted document has an issue, the consultant marks it as needing changes and attaches a message explaining what's wrong (e.g. "photo is blurry, please resubmit") along with a direct WhatsApp contact link, so the student can immediately ask questions and re-upload the corrected document.

## 3.6 Multiple Applications
A student may have multiple concurrent applications open (e.g. applying to more than one university/program at once), all managed under their single assigned consultant.

## 3.7 Full Rejection
If a phase is rejected outright (not just sent back for minor changes), the student can restart that application from the beginning rather than the account being permanently blocked.

# 4. Consultant Flow

- Each consultant logs in with their own account and sees only **their assigned students** — a personal queue, not a shared pool, since each student has exactly one consultant for their entire journey.
- Reviews Phase 1 and Phase 2 documents one student/application at a time; approves, rejects, or requests changes.
- Sends feedback messages tied to a specific status change, including a WhatsApp contact link when a student needs to reach out for clarification or resubmission.
- Marks the Payment Received flag once the office confirms payment was collected, gating progression to full Phase 1 approval.
- Manages **audit documents** — an internal document category attached to a student/application, visible and editable only by the consultant (and admin), never shown to the student. Used for the consultant's own working notes/files on that student; documents can be uploaded, saved for later use, or deleted by the consultant.
- Can browse the same university/program data as students (read-only) to advise students accurately.

---

# 5. Admin Flow

- **Dashboard summary**: at-a-glance counts — applications pending Phase 1 review, pending Phase 2 review, awaiting payment confirmation, and applications approved in the current month.
- Sees **every enrolled/applying student** across all consultants — name, assigned consultant, program, current status.
- **Search** across the student list (by name, phone, or status).
- Manages universities, programs/courses, departments, and scholarships — add, edit, delete.
- Manages **dynamic document requirements** per program and per phase — adding, removing, renaming documents, or marking them mandatory/optional, all from the dashboard without needing a developer to change code.
- Sees the full **activity log** — every status change across every student and consultant (submitted, approved, rejected, changes requested, payment confirmed), with timestamps and the actor who performed each action.
- Assigns and reassigns consultants to students, and can see each consultant's current workload (number of active students) to balance assignments fairly.

# 7. Dynamic Document Requirements

The system holds no hardcoded document lists. Required documents are configured per **program** and per **phase**, editable by admin/consultant from the dashboard:

- **Medical programs (MBBS/BDS/Clinical Medicine):** SSC result card, HSSC result card (or Hope Certificate), ID card copy, 4 white-background photos
- **Engineering programs:** CNIC, passport, photo, SSC & HSSC, physical examination form, non-criminal certificate
- **Chinese Language Course:** CNIC/Form B, passport, photo, SSC & HSSC, bank statement, non-criminal certificate
- **Nursing / Pharmacy:** CNIC/Form B, passport, photo, SSC & HSSC, physical examination form, non-criminal certificate, bank statement
- **Phase 2 (all programs):** passport copy, police clearance certificate, physical examination, 6-month bank statement, introduction video

Admins/consultants can add, remove, rename, or toggle mandatory/optional status on any document, and assign it to a specific phase, without a code change.

---

# 8. Notifications

Two notification layers operate side by side:

1. **System notifications** — automatic and consistently worded: student submits documents → consultant is notified; consultant approves/rejects → student is notified.
2. **Consultant messages** — optional, free-text, attached to a specific status change (especially changes-requested or rejection), explaining exactly what's wrong, with a direct WhatsApp contact button so the student can reach their consultant and resubmit quickly.

**Deadline reminders** are sent to students individually for time-sensitive milestones (e.g. 2nd-year fee due date, visa extension window).

---

# 9. Activity Log

A complete, timestamped audit trail records every status change on every application: submission, review, approval, rejection, changes-requested, payment confirmation, and who performed each action. Visible in full to admin across all students and consultants; consultants see the log scoped to their own assigned students.

# 10. Quality & Reliability Features

- **Document scan quality check** — before an upload is accepted, a basic client-side check flags obviously blurry, cropped, or wrong-format files, reducing avoidable consultant rejections.
- **Weak-connectivity tolerance** — document uploads queue locally and automatically retry on reconnect rather than failing silently, important given some applicants may have unreliable internet access.
- **In-app FAQ / About page** — static content drawn from NRZ's existing company materials: why choose NRZ, office hours, contact details, and a plain-language walkthrough of the application process.
- **Search** — admin and consultant student lists are searchable by name, phone number, or status.
- **Urdu language toggle** — key screens (status, required documents, notifications) available in Urdu in addition to English, given the target audience.

# 11. Performance Requirements

| Concern | Approach |
|---|---|
| Database indexing | Index columns used in filtering/sorting: student ID, consultant ID, program/application type, status, university ID |
| Image optimization | Compress and serve modern formats (e.g. WebP); smaller thumbnails for list views, full resolution only when needed |
| Lazy loading | University/program lists and document previews load incrementally as the user scrolls, not all at once |
| Pagination | Admin's student list, consultant queues, and browse listings fetch in pages (e.g. 20–30 records), never a full table dump |
| Caching | Longer cache duration for rarely-changing data (university/program/scholarship listings) to avoid unnecessary database hits |
| Rate limiting | Basic abuse protection on any custom server-side functions (e.g. notification triggers) |
| Streaming | Reserved for future AI-assisted features (e.g. an eligibility assistant), not required for current scope |

---

# 12. Explicitly Out of Scope

To keep the platform focused and avoid unnecessary complexity, the following are intentionally **not** included:

- **In-app payment processing** — all payments are collected and recorded off-platform between office staff and students; the platform only reflects a manual "payment received" checkpoint, not a payment gateway.
- **Automated document expiry tracking** — consultants review document validity manually during each check; no automated expiry alerts.
- **Parent/guardian portal access** — considered and declined for this version.
- **Firebase or Clerk** — the platform uses a single backend (Supabase: authentication, database, and file storage together) rather than introducing a second identity or data platform.
- **Message queues, load balancers, or a dedicated caching layer (e.g. Redis)** — not required at current scale; the managed backend already handles connection pooling and storage. These may be revisited only if usage grows substantially.
- **In-app real-time chat** — consultant-to-student contact is handled via a WhatsApp deep link, not a built-in messaging system.
- **Multi-branch support** — the platform is scoped to NRZ's single Rawalpindi office for this version.

# 13. Open Items for Future Consideration

- **QR code deep linking** — NRZ's printed brochures include a "scan to download brochure" QR code. A future enhancement could point this directly into the platform at the relevant university/program's application screen, including for users who don't yet have an account (deferred deep linking), pending evaluation of a suitable linking service.
- **Referral tracking** — an optional "referred by" field at signup, to help admin understand word-of-mouth growth.
- **Multi-branch expansion** — if NRZ opens additional offices, applications and consultants would need to be scoped per branch.
